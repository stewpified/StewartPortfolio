Recipe & Crafting System

A DataTable-driven crafting system that lets new recipes be added by editing rows, not Blueprint graphs.

Blueprints Data Tables Systems
screenshot / clip

The Problem

Hardcoding each recipe directly into Blueprint logic doesn't scale — every new recipe means editing graph logic by hand, and there's no clean way to validate a player's inventory against a recipe's ingredient list without repeating that logic per recipe.

Approach

  • Defined FRecipeIngredient and FRecipeRow structs so each recipe becomes a single DataTable row: an ingredient list (item ID + quantity), an output item, and a required station.
  • Built TryCraftRecipe as a single validation function that checks the player's inventory against a recipe's ingredient list, handling multiple ingredients and partial stacks.
  • Added station-gating through an EStationType enum, so a recipe only surfaces at the station it's meant for (a soup recipe won't show up at a workbench).
  • Cached the list of currently-craftable recipes and only recalculated it when the inventory actually changed, instead of re-validating every recipe on every UI refresh.

Breakdown

// DataTable row shape (Blueprint structs) Struct FRecipeIngredient ItemID Quantity Struct FRecipeRow DisplayName Ingredients [array of FRecipeIngredient] OutputItem RequiredStation // EStationType // crafting flow TryCraftRecipe(Recipe) → for each Ingredient in Recipe.Ingredients: check Inventory has enough Quantity if all pass → consume ingredients, grant OutputItem

Outcome

New recipes can be added purely through DataTable rows, with zero Blueprint graph edits. The cached craftable-recipe list keeps the crafting UI responsive even as the recipe book grows.
